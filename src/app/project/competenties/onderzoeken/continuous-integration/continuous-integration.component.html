Onderzoek Continuous Integration

Inleiding

Dit onderzoek schrijf ik om de processen binnen ons tech bedrijf te kunnen optimaliseren. Momenteel worden er nog (te) veel fouten gemaakt met het live zetten of updaten van projecten. Deze workflow is momenteel volledig afhankelijk van menselijke handelingen en deze brengen ook menselijke fouten met zich mee. Deze fouten en de risico's die dit met zich meebrengen wil ik beantwoorden met de volgende onderzoeksvraag:


"Hoe kan het risico op menselijke fouten binnen een "Continuous Integration" omgeving worden geminimaliseerd?"


  • Onderzoeksaanpak

    Mijn onderzoek zal bestaan uit het uitzoeken van de problemen die ondervonden worden bij mijn stagebedrijf met het live zetten van nieuwe projecten en updates uitrollen voor bestaande projecten. Hoe deze problemen aangepakt kunnen worden, welke methodes er gebruikt worden en een stuk deskresearch om deze mee te vergelijken en een passende keuze te kunnen maken voor het bedrijf.


    Het probleem

    Om te beginnen een illustratie die de huidige situatie aardig opsomt



    Over het algemeen gebeuren de meeste fouten als er updates uitgerold worden. Als een project eenmaal opgezet is dan draait het over het algemeen goed. Een developer schrijft zijn code op zijn eigen computer, test hier ook op en zet zijn wijzigingen door naar de server. Het kan zijn dat hij niet alle bestanden uploadt, of dat er conflicten in bestanden ontstaan of zelfs hele stukken code per ongeluk verwijderd waar een collega aan gewerkt heeft.

    De professionals zeggen

    Mojtaba Shahin, Muhammad Ali Babar en Liming Zhu (2017) schrijven het volgende: “We have identified 30 approaches and associated tools, which facilitate the implementation of continuous practices in the following ways: (1) reducing build and test time in continuous integration (CI); (2) increasing visibility and awareness on build and test results in CI; (3) supporting (semi-) automated continuous testing; (4) detecting violations, flaws, and faults in CI; (5) addressing security and scalability issues in deployment pipeline; and (6) improving dependability and reliability of deployment process.”.

    Dit somt in grote delen de punten op die Bol & Boosch als problemen ervaart. Mede door aanbevelingen van deze heren ben ik tot de beschikking gekomen van een set tools, zoals Github en Jenkins, waarmee ik verder kon in mijn zoektocht. Om er achter te komen wat de beste tools zijn voor mijn stage bedrijf ben ik deskresearch gaan doen.


    Deskresearch

    In dit gedeelte ga ik in op verschillende tools waarmee de problemen van mijn stagebedrijf verlicht kunnen worden. Hierin refereer ik een aantal keer naar “Git”. Git is een vorm van versiebeheer waarin ontwikkelaars hun code kunnen opslaan. Hierin bestaan meerdere “branches. Deze branches refereren aan de staat waarin dat specifieke stuk code zich bevindt. Develop staat voor in ontwikkeling, staging is klaar om getest te worden en master is stabiele code.

    Doel

    Het doel van dit onderzoek is om een tool te vinden waar developers geholpen worden met het testen en deployen van hun applicaties en/of websites. Het idee is om developers van de staging en productie servers af te houden (lees: zelf niet meer uploaden via FileZilla). Door een push op de develop en master branch wordt het project automatisch gebouwd door een pipeline die ingesteld staat op de git server.


    Dit is de traditionele (handmatige) aanpak, zoals het momenteel bij Bol & Boosch gaat.

      Traditional Approach
    • Code Check-in
    • Compile and BuildDeployment Archive
    • Approvals
    • Request for deployment
    • Testing

    Dit is het gewenste resultaat, geautomatiseerd.

    Jenkins vs Gitlab-CI

    In dit onderdeel ga ik twee tools analyseren, zodat ik de voor- en nadelen tegenover elkaar kan zetten.


    Het gaat om Jenkins en Gitlab-CI, allebei tools waarmee het mogelijk is om ons doel te behalen.


    Jenkins

    Deze tool bestaat al sinds 2011 en is gebouwd in java. Het grote voordeel is dat Jenkins geleverd wordt met een gui (web interface). Het is opensource en er zijn honderden plugins beschikbaar om onze wensen te verwezenlijken. Echter werkt het niet out of the box en vereist het werk om het systeem in te stellen. Het werkend krijgen van de service is met één terminal commando uit te voeren, dat is prettig.




    Jenkins wordt door de community vaak als een van de betere opties aangeraden.


    Gitlab-CI

    Het grote voordeel van GitLab-CI is dat het geïntegreerd zit in GitLab. Dit is toevallig ook de partij waar de Bol & Boosch developers hun repositories onderbrengen. Je hebt niet echt een GUI hebt waar je op terug kan vallen, je kan wel de huidige jobs en pipelines bekijken. Het instellen van de repo’s gaat echter wel via een configuratie bestand, het is daarom niet overzichtelijk voor de leek. Ook mist Gitlab-CI plugins, zoals Jenkins deze heeft. Dit zou eventuele uitbreidingen kunnen hinderen.


    Een groot voordeel is dat er geen installatie hoeft te gebeuren, het wordt meegeleverd met GitLab.

    Eigen bevindingen

    Gitlab-CI stopte al heel snel met loggen als er bijvoorbeeld een job voor de kennisbank opengezet wordt. Dit is omdat het maximaal aantal regels bereikt is. Dit kan voor sommige applicaties best lastig worden, aangezien er veel modules gebruikt kunnen worden. Helaas is Gitlab-CI qua git aanbieders beperkt tot Gitlab.

    Jenkins daarentegen werkte vlekkeloos, maar kost wat meer moeite om in te stellen. De user interface is prettig en overzichtelijk. Jenkins is niet beperkt tot een aanbieder van git oplossingen.




    Conclusie

    Balalaie, Heydarnoori, en Jamshidi (2016) zeiden in hun onderzoek:

    “DevOps is a set of practices that aim to decrease the time between changing a system and transferring that change to the production environment. However, they also insist on maintaining software quality in terms of both code and the delivery mechanism.”


    Door de bevindingen van professionals en testen die er gedaan zijn tijdens het deskresearch kan ik concluderen dat de onderzoeksvraag beantwoord is en kan Bol & Boosch aan de slag met de bouw van haar eigen Continuous Integration omgeving. Door een development en productie omgeving naar deze standaarden in te richten worden er minder fouten gemaakt en is het uitgeven van updates van bestaande en uitrollen van nieuwe projecten veiliger, getest en makkelijker.


    Dit scheelt niet alleen veel werk, maar ook het moraal van de werknemers kan verbeteren. Eerder in het stuk zei ik het volgende: “Op dit soort momenten voel je jezelf soms meer een "brandweerman" dan een daadwerkelijke developer. Je bent constant brandjes aan het blussen.” Met een oplossing zoals deze kunnen developers doen waar ze goed in zijn: Het ontwikkelen van producten.
